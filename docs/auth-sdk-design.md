# SDK авторизации: проектирование

## Цель
Сделать клиентскую библиотеку для интеграции авторизации в сервисы, которая:

- минимизирует ручной код в прикладных сервисах;
- работает с auth-service через gRPC;
- поддерживает проверку прав с учетом области действия;
- легко подключается к Flask и FastAPI;
- допускает гибкую деградацию при недоступности auth-service.

## Контекст

### Что уже есть
- Auth-service хранит роли и атомарные права.
- Роли собирают атомарные права в более крупные наборы.
- JWT не должен нести бизнес-данные, только `user_id`.
- Токены для других сервисов выдаются auth-service.
- Нужна возможность межсервисной авторизации без `user_id`.
### Что нужно от SDK
- Удобный API для проверок прав.
- Поддержка scope на уровне роли: scope ограничивает permissions роли по области применения.
- Интеграция через декораторы и зависимости.
- Возможность использовать локальный кэш, но итоговая модель должна оставаться централизованной.

## Предлагаемая модель

### Основные сущности

#### Permission
Атомарное право, например:
- `user.read`
- `user.update`
- `program.manage`

#### Role
Набор атомарных прав. Роль нужна для удобного назначения и композиции.

#### Scope
Область действия права. Идея scope нужна для случаев вида:
- право на конкретное подразделение университета;
- право на конкретную образовательную программу;
- в будущем любые другие объектные ограничения.

Практически scope лучше не связывать жёстко с одной сущностью. Более гибкий вариант:
- `scope_type` — тип области действия;
- `scope_id` — идентификатор конкретного объекта;
- опционально дополнительные атрибуты, если они понадобятся позже.

Пример:
- `department:42`
- `program:17`
- `entity:user:123`

Такой формат проще расширять, чем отдельные поля под каждый бизнес-случай.

## Рекомендуемый подход к проверке прав

### Базовая идея
SDK не должен сам вычислять права “по памяти” как единственный источник истины. Лучше сделать так:

- SDK получает контекст пользователя или сервисной идентичности;
- формирует запрос в auth-service;
- auth-service решает, разрешено действие или нет;
- SDK может кэшировать результат, если это допустимо политикой.

### Почему это хорошо
- единая точка принятия решений;
- проще менять модель прав без обновления всех сервисов;
- меньше риск рассинхронизации между сервисами;
- удобнее вводить audit и трассировку решений.

## Scope-модель

### Что важно решить
Нужно заранее определить, как SDK будет описывать проверку вида:

- “пользователь может редактировать программу 17”; 
- “пользователь может смотреть только свои сущности”; 
- “пользователь может работать только в подразделении X”.

### Практичный формат
Лучше привязать scope к роли, а не к отдельному permission.

Это значит, что у роли может быть:

- глобальный набор permissions;
- scoped-набор permissions для конкретной области;
- и при этом часть permissions может пересекаться между глобальной и scoped-версией роли.

Смысл scoped-ограничения такой: scope не создает новый permission, а только ограничивает, где уже существующий permission действует.

Если у роли есть глобальный scope, он должен иметь приоритет над локальным scope и перекрывать его. Это нужно, чтобы условная роль admin давала все свои permissions на все области применения.

Логика объединения должна быть такой:

- если у роли есть глобальный scope, ее permissions действуют во всех scope и не ограничиваются локальными scope;
- если scoped-роль и глобальная роль содержат один и тот же permission, в итоге остается глобальное значение этого permission;
- если permission есть только в scoped-роли и отсутствует в глобальном наборе, он действует только в пределах scoped-role scope;
- если две роли дают один и тот же permission, а одна из них глобальная, глобальный scope перезаписывает локальный;
- если scopes разные и ни один из них не глобальный, итоговый permission действует в объединенном наборе scopes для этого permission.

Такой подход позволяет:

- не раздувать число отдельных permissions под каждый scope;
- использовать одну и ту же роль как в глобальном, так и в scoped-варианте;
- явно хранить ограничения по области применения на уровне роли, а не на уровне каждого permission;
- при наличии нескольких ролей получать объединенный доступ по каждому permission, а global-role автоматически давать доступ без ограничений.

Практически это можно представить как роль, у которой есть два слоя:

- общий слой permissions;
- scoped-слой ограничений, который уточняет, в каких scope каждый permission действует.

Если у пользователя есть две роли с разными scope, то для каждого permission нужно собирать объединенный набор допустимых scope. Иначе говоря, permission из нескольких ролей складывается не только по факту наличия, но и по области действия.

## API SDK

### Базовый уровень
Нужен компактный API вроде:

- `check(permission, scope=None)`;
- `require(permission, scope=None)`;

`scope` должен уметь передаваться двумя способами:

- через `ScopeResolver`, если SDK или интеграция умеет извлечь его из входящего запроса;
- вручную, если в проекте нет единого способа парсинга запроса или нужный scope вычисляется внутри handler’а.

### ScopeResolver
Чтобы не превращать декораторы и зависимости в набор произвольных callback’ов, лучше ввести отдельную маленькую абстракцию `ScopeResolver`.

`ScopeResolver` отвечает только за получение `Scope` из request, path params, body, dependency context или другого окружения проекта. Он не принимает решение авторизации и не знает бизнес-логику handler’а.

Это дает несколько преимуществ:

- одинаковый интерфейс для Flask и FastAPI;
- проще тестировать извлечение scope;
- меньше магии в декораторах;
- можно переиспользовать resolver’ы между разными сервисами.

Примерно `Scope` может выглядеть так:

```python
Scope(scope_type="department", scope_id="42")
```

### Для Flask и FastAPI
SDK может дать два уровня интеграции:

- низкоуровневые методы для явного вызова внутри handler’ов;
- высокоуровневые декораторы и зависимости для типовых проверок.

Идея такая: если проект позволяет, интеграция получает `scope_resolver` и использует его для извлечения `Scope` из request / dependency context. Если resolver’а нет или он не подходит, handler должен иметь возможность явно передать `Scope` в вызов проверки.

Для более удобного сценария декоратор или dependency может принимать `scope_resolver`. Тогда flow будет выглядеть так:

- декоратор или dependency проверяет базовое право;
- `ScopeResolver` извлекает `Scope` для текущего запроса;
- `AuthContext` получает этот scope и может использовать его для `check_scope(...)`;
- если проекту удобнее, scope можно передать вручную прямо в body handler’а.

Для FastAPI логичны зависимости.
Для Flask удобнее декораторы.

## Интеграция с Flask

### Желательный стиль
- декоратор на endpoint;
- передача контекста пользователя из `request`;
- явное указание нужного права или набора прав;
- возможность передать `scope` как аргумент, вычисляемый из параметров маршрута;
- возможность вообще не полагаться на автоматический парсинг и задать scope вручную в теле обработчика.

Примерно это может выглядеть так:

```python
@required("user:update", scope_resolver=DepartmentScopeResolver())
def update_user_model(auth_context: AuthContext, ...):
    auth_context.check_scope(scope_type="department", scope_id=scope_id)
```

Где `DepartmentScopeResolver` знает, как достать `scope_id` из request, path params или другой части окружения.

### Примерный сценарий
- получить `user_id` из токена;
- из route params взять `department_id` или `program_id`;
- вызвать check через SDK;
- при отказе вернуть `403`.

## Интеграция с FastAPI

### Желательный стиль
- dependency, который проверяет права;
- возможность передавать `scope` через dependency factory;
- возможность переиспользовать один и тот же auth-контекст в нескольких зависимостях.

Если проект не хочет или не может парсить scope из dependency context, должна оставаться возможность передать scope вручную в сам вызов проверки.

Для FastAPI это может выглядеть аналогично Flask: dependency принимает `scope_resolver`, который умеет вычислять `Scope` из `Request`, `Depends`, path params или body, а дальше передает результат в проверку.

### Примерный сценарий
- dependency извлекает `user_id` из security context;
- dependency получает scope из path params;
- SDK выполняет check;
- при отказе выбрасывается HTTP exception `403`.

## gRPC против HTTP

### Почему gRPC выглядит сильным кандидатом
- компактный контракт и строгая схема сообщений;
- удобнее для внутреннего service-to-service взаимодействия;
- обычно лучше для частых коротких запросов проверки прав;
- проще строить единый typed SDK под разные языки, если это когда-то понадобится.

### Когда HTTP может быть полезен
- проще ручная отладка;
- легче проверять через curl / браузер / proxy;
- проще временно интегрировать внешние сервисы без полноценного gRPC-стека.

### Практический вывод
Если auth-service уже внутренний и проверки частые, gRPC выглядит базовым транспортом. HTTP можно оставить как дополнительный административный или debug-интерфейс, но не обязательно делать его основным.

## Гибридная схема

### Рекомендация
Сейчас лучше исходить из гибридной модели:

- auth-service остается источником истины;
- SDK может держать короткий локальный кэш для уменьшения latency;
- при этом критические решения должны опираться на актуальные данные и ясную политику инвалидation.

### Что нужно заранее определить
- срок жизни кэша;
- что кэшируется: итог решения, список прав, список ролей или всё вместе;
- как происходит сброс после изменения ролей;
- что делать при недоступности auth-service.

## Поведение при сбоях

SDK должен поддерживать настраиваемую политику:

- deny-by-default для чувствительных операций;
- cached-last-known для снижения простоя;
- configurable per route / per service, если для разных операций нужна разная строгость.

Для авторизации обычно безопаснее deny-by-default, а отклонения от этого должны быть явными.

## Наблюдаемость

Для такого SDK полезны:

- метрики количества проверок;
- метрики latency по auth-service;
- доля cache hit / miss;
- причины отказа;
- correlation id / trace id для разбора инцидентов.
